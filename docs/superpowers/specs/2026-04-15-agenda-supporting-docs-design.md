# Agenda Supporting Documents — Design Spec

**Date:** 2026-04-15
**Status:** Approved

---

## Goal

Fix the council agenda scraper so it downloads only the **separately-uploaded supporting documents** (not agenda-embedded attachments), associates each document with its parent agenda item, names files with an `Item X.Y` prefix, optionally produces a combined PDF with a cover-page index, and writes a markdown-formatted run summary after each run.

---

## Background & Key Insight

The CivicClerk agenda viewer shows two categories of linked files per agenda item:

1. **Agenda-embedded documents** — labelled `Report`, `Attachment A`, `Attachment B`, etc. These are sections *inside* the agenda PDF itself. Clicking them navigates to that section of the document. They do **not** need to be separately downloaded.

2. **Separately-uploaded supporting documents** — labelled `Supporting Document N - Title` (older agendas) or descriptive names like `Council Member Memo 175th Anniversary` (newer agendas). These are independent files uploaded alongside the agenda. These **are** what we want to download.

The previous implementation incorrectly treated `Attachment X` labels as supporting documents. This spec corrects that.

---

## Document Detection — Approach A (Exclusion-based)

**Excluded from download (agenda-embedded):**
- Exact matches (case-insensitive): `report`, `information only report`, `corro / resolution report`, `cover page`
- Any text starting with `attachment` (covers `Attachment A`, `Attachment B Libraries`, `Attachment A - Title`, etc.)

**Included for download (separately-uploaded):**
- Everything else — `Supporting Document N - Title`, `Supplementary ...`, descriptive names like `Council Member Memo 175th Anniversary`, and any future naming conventions the council introduces

**Rationale for Approach A:** An inclusion-based approach (matching `Supporting Document`, `Supplementary`, etc.) would silently miss documents when the council changes naming conventions — as happened between the September 2025 and April 2026 agendas. An exclusion-based approach only needs updating if the council introduces a *new embedded-document label*, which is far less common. This design choice must be documented in the README.

---

## Section Scoping

Supporting documents are only extracted from items within these major agenda sections. All other sections are ignored.

**Primary (always searched):**
- Decision Items
- Motions on Notice
- Information Items
- Information Only Reports

**Secondary (less common but included):**
- Response to Gallery Questions from Previous Meetings
- Response to Questions on Notice from Previous Meetings
- Questions on Notice
- Items Without Notice
- Motions Without Notice
- Questions Without Notice

Matching is case-insensitive. The analyser tracks the `current_section` as it processes HTML cells in order. When a major-item cell is encountered, it sets `current_section` if the cell text matches a target section name; otherwise `current_section` is cleared. Links are only extracted when `current_section` is set.

---

## Item Association

The CivicClerk AIDocs HTML alternates between:
- **Non-preview cells** (`dx-wrap dxtl dxtl__B0`) — contain the agenda item text
- **Preview cells** (`dxtlPreview_CustomThemeModerno dxtl__B0`) — contain the GenFile.aspx links for the preceding item

Processing cells in order, the analyser tracks `current_individual_item`. When a preview cell is reached, its qualifying links are tagged with `parent_item_number` (e.g. `"10.1"`) and `parent_item_text` (e.g. `"10.1 City of Mitcham 175th Anniversary Publication Project"`).

If an item has only a major number (no sub-item, e.g. `"10"`), the prefix becomes `Item 10`.

---

## File Naming

```
Item {X.Y} {Title}.pdf
```

Examples:
- `Item 10.1 Council Member Memo 175th Anniversary.pdf`
- `Item 11.1 Supporting Document - Behavioural Standards.pdf`

Rules:
- Strip the `"Supporting Document N - "` prefix if present (already done in existing code)
- Remove filesystem-unsafe characters (`< > : " / \ | ? *`)
- Collapse multiple spaces to one
- Cap **total filename** (including `Item X.Y ` prefix and `.pdf` extension) at 120 characters — truncate the title portion, never the prefix or extension
- `ad_value`-based `SD_XXXX_` prefix is **removed** — replaced entirely by the `Item X.Y` prefix

## Combined PDF (`--combine` flag)

New module: `pdf_combiner.py`

**Dependencies:** `pypdf` (merge), `reportlab` (cover page generation)

**Output file:** `{Meeting Title} - Supporting Documents.pdf` in the meeting folder

**Structure:**
1. **Cover page** (generated by ReportLab):
   - Meeting title, run date
   - Table of contents: Item number | Document title | Page number
2. **Merged content:** all downloaded PDFs concatenated in the order they appear in the HTML (i.e. by item, then by document position within that item)

**Interface:**
```python
def combine_pdfs(
    pdf_entries: list[dict],   # [{'filepath', 'item_number', 'title'}, ...]
    output_path: str,
    meeting_title: str,
    run_date: str,
) -> str:                      # returns output_path on success
```

If fewer than 2 PDFs are available to merge, skip silently and log a warning.

## Run Summary

After each run, write `_run_summary.txt` into the meeting output folder. Content is markdown-formatted (readable as plain text; renders correctly as markdown on GitHub).

**Content:**
```markdown
# {Meeting Title} — Supporting Documents Run

**Run date:** YYYY-MM-DD HH:MM
**URL:** https://...

## Documents

| Item | Title | Status | File |
|------|-------|--------|------|
| 10.1 | Council Member Memo 175th Anniversary | Downloaded | Item 10.1 Council Member Memo....pdf |

## Summary
- Supporting documents found: N
- Downloaded: N  Failed: N
- Combined PDF: {filename} (or "Not generated — use --combine to create")
```

The summary is written even if zero documents were found (so a run that found nothing is still recorded).
